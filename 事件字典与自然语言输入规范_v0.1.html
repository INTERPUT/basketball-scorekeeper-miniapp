1. 范围与原则

首版 根据产品文档 v0.7，首版只覆盖正式计分表能力，不把篮板、助攻、抢断、盖帽、失误等扩展技术统计纳入 MVP。

1.1 首版纳入范围

  • 得分事件：一分、两分、三分。
  • 犯规事件：个人犯规、球队犯规自动累计；单节第 5 次及之后犯规触发罚球提醒。
  • 暂停事件：球队暂停；上半场 2 次、下半场 3 次、每个加时 1 次。
  • 比赛控制事件：启动计时、暂停计时、继续计时、更正当前时间、手动结束单节、结束比赛。
  • 纠错事件：撤销上一条、修改指定事件、补录历史事件。

1.2 设计原则

  • AI 只生成标准事件草稿，不直接写入正式数据。
  • 凡是会改变正式比分、犯规、暂停或历史记录的动作，都必须由技术员确认。
  • 事件以“事件流”保存，导出结果由事件回放生成，不直接手改汇总值。
  • 若自然语言缺少关键字段，系统直接退回，不进行对话式补问。
  • 同一事件只落一次账；重复输入必须可识别并让技术员确认。

2. 标准事件模型

2.1 每条事件的基础字段

字段 是否必填 说明
event_id 是 系统生成的唯一事件 ID。
event_type 是 标准事件类型，例如 score、personal_foul、timeout。
team_id 视事件而定 所属球队；得分、犯规、暂停必填。
player_id 视事件而定 球员相关事件必填；球队暂停可为空。
period 是 当前节次。
game_clock 是 事件发生时的比赛时间。
source_text 是 原始文本，若来自语音则保存转写结果。
source_mode 是 text 或 voice。
status 是 draft、confirmed、corrected、reverted。
created_by 是 首版通常为当前技术员。

2.2 标准输出建议

{
  "event_type": "score",
  "team": "白队",
  "player_no": "7",
  "score_value": 2,
  "period": 1,
  "game_clock": "08:21",
  "data_effect": {
    "team_score_delta": 2,
    "player_score_delta": 2,
    "append_running_score": true
  },
  "needs_confirmation": true,
  "ambiguity": []
}

罚球组结果 free_throw_series_result 球队、球员、罚球次数、命中次数、节次、时间 按命中次数增加球队与球员得分；例如 2罚中0 记 0 分，2罚中1 记 1 分。 白队 7 号 2 罚中 0、白队 7 号 2 罚 0 中

4. 自然语言输入规范

4.1 推荐语序

推荐 最稳定的表达顺序是：球队 + 球员 + 事件。例如：白队 7 号两分命中。

4.2 可接受表达

标准事件 可接受表达 归一化结果
两分命中 白队7号两分命中、白队7号进了个两分、白队7号两分 score_two_point
三分命中 蓝队12号三分命中、蓝队12号投进三分 score_three_point
罚球命中 白队7号罚球命中、白队7号罚进一个 score_free_throw
罚球组结果 白队7号2罚中0、白队7号2罚0中、白队7号2罚中1 free_throw_series_result
个人犯规 蓝队12号犯规一次、蓝队12号个人犯规 personal_foul
球队暂停 白队暂停、白队叫暂停 team_timeout
计时控制 开始计时、暂停计时、继续计时、把时间改成03:12、结束本节比赛 clock_start / clock_pause / clock_resume / clock_correct / period_end

4.3 必须明确的字段

  • 得分事件必须明确：球队、球员、分值类型。
  • 罚球组结果必须明确：球队、球员、罚球次数、命中次数。
  • 个人犯规必须明确：球队、球员。
  • 球队暂停必须明确：球队。
  • 比赛进行中使用 暂停计时，暂停状态下使用 继续计时；更正当前时间 必须明确目标时间。
  • 补录历史事件必须额外明确：节次、比赛时间。
  • 修正历史事件必须明确：要改哪一条、改成什么。

4.4 系统可自动补充的字段

  • 当前节次：若系统已知，可自动带入。
  • 当前比赛时间：若为实时输入，可自动带入。
  • 球队名称别名：若赛前已配置，例如“白队”对应“高一三班”。

统一规则 技术台人员经过基础培训。若输入不是完整标准记录，系统统一返回： 缺少球队 / 球员 / 事件，请重新给出完整技术记录。

白队得分 缺球员、缺分值 返回：缺少球队 / 球员 / 事件，请重新给出完整技术记录。
蓝队犯规 缺球员 返回：缺少球队 / 球员 / 事件，请重新给出完整技术记录。

6. 更正与撤销规则

6.1 撤销

  • 撤销上一条 默认只作用于最近一条已确认事件。
  • 撤销前必须展示将被撤销的事件摘要。
  • 撤销后原事件状态改为 reverted，不能物理删除。

6.2 修改

  • 修改不是覆盖原事件，而是保留原事件并新增一条修正记录。
  • 系统应展示“原事件 → 新事件”的差异。
  • 修改比分事件时，需要重算球队总分、球员得分、逐分记录和逐节比分。

6.3 补录

  • 补录必须提供明确节次和时间。
  • 补录后按事件时间插入历史时间线，并触发重新汇总。
  • 若补录会影响已经签字的结果，必须生成新版本，不得覆盖旧版。

6.4 规则提醒

  • 球员个人犯规达到 5 次时，生成 player_foul_out_alert 提醒。
  • 球队单节累计 4 次犯规后，第 5 次及之后犯规生成 team_bonus_alert 提醒，提示需要罚球。
  • 比赛剩余时间进入 01:00 时，生成 last_minute_alert 提醒。

7. 语音输入规范

已定 首版语音识别使用火山引擎。语音只负责转文本，事件理解仍走与文本输入相同的 AI 解析链路。
  • 必须保留语音转写文本，让技术员在确认前可手动修改。
  • 如果转写文本不完整或明显异常，不得直接生成已确认事件。
  • 建议保留原始音频引用或识别结果元数据，便于问题排查。
  • 语音输入失败时，应降级到文本输入，不阻断比赛记录。

8. 暂不支持范围

首版暂不纳入

  • 篮板
  • 助攻
  • 抢断
  • 盖帽
  • 失误
  • 投篮出手与命中率

处理原则

若用户输入上述事件，系统应提示： 首版暂不记录该类扩展技术统计。 不应把它们误记成正式记分事件。

白队 7 号 2 罚中 0 free_throw_series_result 生成草稿：罚球 2 次、命中 0 次、加 0 分，等待确认。

白队暂停 team_timeout 生成草稿：按当前阶段扣减暂停次数并记录暂停。
7 号得分 无法确定 返回标准退回提示，不生成事件。
