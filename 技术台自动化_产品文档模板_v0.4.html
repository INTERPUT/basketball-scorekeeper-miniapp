已定 以下内容已经根据最新批注确定，后续除非需求变化，不再作为开放问题反复讨论。
决策项 当前结论 影响
目标用户 首版服务学校班赛，不以正式协会赛事为目标。 规则配置保持简洁，避免过早引入复杂赛制。
首版范围 先实现现有正式计分表能力，不在第一版追求完整高级技术统计。 MVP 聚焦得分、犯规、暂停、比赛计时控制、逐节比分、逐分记录、规则提醒、签字留档。
技术台人数 首版按单人技术员操作设计。 界面要适合单人快速录入，默认不做多人协作。
观众端 观众与技术台共用同一套比赛信息页面，但观众只读。 降低页面复杂度，统一比分与统计展示逻辑。
最终输出模板 采用当前图片中的正式计分表作为首版最终输出模板。 导出文件以可复现该表格内容为硬约束。
实现平台 首版以微信小程序实现。 交互和分享方式围绕小程序场景设计。
输入方式 首版支持文本输入和语音输入；语音识别 API 采用火山引擎，转成文本后进入同一解析链路。 降低比赛中快速录入门槛。
进入方式 观众通过创建房间时设置的四位房间码进入比赛页。 首版不依赖公开链接或复杂分享链路。
签字范围 赛后仅需双方队长签字。 首版签字流程保持轻量。
名单复用 首版不做名单导入，采用手动录入或复制粘贴文本完成名单填写。 保持首版简单，名单导入后置到后续阶段。
房间生命周期 房间码固定为四位，创建房间时设置；比赛结束后房间销毁。 控制房间数量，避免赛后残留和重复误入。
访问与权限 观众匿名只读；创建比赛的技术员基于微信身份自动成为唯一 operator。 保持加入门槛低，同时防止非创建者误改正式比赛数据。
网络能力 首版不做离线续记，只提供断网提示。 优先保证在线场景正确，避免首版引入事件冲突合并复杂度。
高级信息 创建比赛页采用“基础信息 + 可折叠高级信息”，高级信息包含赛事名称、地点、裁判等正式表头字段。 不拖慢开赛，同时保证正式导出字段完整。
归档保留 房间销毁后，仍保留完整结构化归档和导出文件。 支持复盘、重新导出和后续更正。

3. 产品概述

要解决的问题

  • 学校班赛中，技术台通常依赖少量熟手，普通同学难以上手。
  • 纸质记录速度慢、容易漏记，赛后整理和归档重复劳动多。
  • 观众和球员无法稳定看到实时比分、犯规和逐节信息。
  • 比赛结束后仍需手工汇总、签字、生成留底文件。

产品目标

  • 让一名普通技术员经过短时间熟悉后即可完成整场比赛记录。
  • 让自然语言成为主录入方式，减少对专业术语和复杂按钮的依赖。
  • 让比分、犯规、暂停、计时和逐分过程实时同步并可纠错。
  • 让赛后标准表格、报告和签名留底自动生成。

3.1 产品原则

  • 先保证正式记分可靠，再考虑扩展统计。
  • AI 负责理解与建议，技术员负责确认与生效。
  • 比赛数据必须基于事件流保存，便于撤销、补录、改判和导出。
  • 技术员和观众共用同一套比赛信息模型，仅权限不同。
  • 所有正式输出都必须可回溯到原始事件和签字版本。

4. 用户与权限

角色 主要权限 首版要求
技术员 创建比赛、配置阵容、录入事件、确认修改、控制计时、导出结果 首版唯一编辑角色。
观众 / 球员 查看比分、节次、时间、犯规、暂停、逐节与逐分信息 只读；与技术员共用相同比赛信息页面。
队长 赛后查看结果并完成双方队长签字确认 只在赛后进入签字流程。

5. 范围定义

5.1 MVP 范围

模块 首版必须支持
比赛创建 创建房间、设置四位房间码、球队、球衣颜色、球员姓名与号码、单节时长。
正式记分 得分、个人犯规、球队犯规、暂停、比赛计时控制、逐节比分、逐分记录、最终比分。
自然语言录入 支持文本和语音输入；语音先经火山引擎语音识别 API 转成文本，再由 AI 解析为标准事件，技术员确认后生效。
比赛展示 观众通过四位房间码进入同一比赛页面，查看总比分、节次、剩余时间、犯规、暂停和相关正式统计。
赛后留档 双方队长签字、导出 PDF、导出正式计分表、保留结构化数据。

5.2 后续阶段

第二阶段

增强扩展技术统计、更多快捷按钮、历史比赛检索、阵容复用。

第三阶段

平台化多人协作、赛程管理、排行榜、自动战报、数据分析。

5.3 非目标

  • 首版不以正式协会赛事的完整复杂规则为目标。
  • 首版不默认支持多名技术员同时编辑。
  • 首版不要求完整职业级技术统计体系。
  • 首版不做名单导入和名单复用能力。
  • 首版不做复杂赛事运营后台。

6. 核心流程

6.1 赛前

  1. 技术员创建比赛房间。
  2. 录入双方球队、球衣颜色、球员姓名和号码。
  3. 设置单节时间和基础比赛参数。
  4. 创建四位房间码，观众和球员输入房间码加入查看。

6.2 赛中

  1. 技术员输入或说出自然语言，例如：白队 7 号两分命中。
  2. 若使用语音输入，系统先通过火山引擎语音识别 API 转写为文本。
  3. AI 解析为标准事件卡片，并展示其会造成的数据变化。
  4. 技术员确认后，系统更新比分、犯规、暂停、计时状态和逐分记录。
  5. 当球员个人犯规达到 5 次、球队单节第 5 次及之后犯规、比赛仅剩 1 分钟时，系统主动提醒。
  6. 共用比赛页面实时同步给观众。
  7. 若发生误记，技术员可撤销、补录或修正历史事件。

6.3 赛后

  1. 系统自动汇总整场结果。
  2. 双方队长在屏幕上完成电子签名。
  3. 系统导出正式计分表、PDF 报告和结构化数据。
  4. 比赛归档，房间销毁；后续若需更正则生成新版本。

7. 功能需求

ID 模块 需求 优先级 验收口径
FR-01 比赛创建 技术员可创建比赛房间并录入基础信息。 P0 可成功生成一场可进入的比赛。
FR-02 阵容管理 可录入球队名称、球衣颜色、球员姓名和号码。 P0 双方名单可在比赛页与导出表中正确显示。
FR-03 比赛规则 可设置单节时长，并允许比赛中途修改。 P0 修改后后续计时按新设置运行，并保留修改记录。
FR-03A 比赛计时 支持启动计时、暂停计时、继续计时、更正当前时间、手动结束本节比赛。 P0 技术员可在比赛中控制时钟，且所有更正有记录。
FR-04 自然语言录入 系统可将文本输入转成标准事件草稿。 P0 输入常见记分语句后，可返回明确的标准事件内容。
FR-05 确认机制 正式数据只在技术员确认后变更。 P0 未确认事件不改变比分或正式统计。
FR-06 正式记分 支持得分、犯规、暂停、逐节比分、逐分记录；暂停规则为上半场 2 次、下半场 3 次、每个加时 1 次。 P0 导出结果与比赛过程一致。
FR-06A 规则提醒 球员个人犯规达到 5 次时提醒；球队单节累计 4 次后，第 5 次及之后犯规提示需要罚球；比赛仅剩 1 分钟时提醒。 P0 关键规则节点在技术台页可见，且不依赖技术员自行计算。
FR-07 只读观看 观众输入有效四位房间码后，可进入同一比赛页面查看实时信息。 P0 观众无编辑入口，看到的数据与技术员页一致。
FR-08 纠错 支持撤销、改判、补录，并保留事件日志。 P0 修正后可以追溯是谁在何时做了何种修改。
FR-09 赛后签字 支持双方队长在设备上完成电子签字。 P0 双方队长签名可进入最终留底文件。
FR-10 导出 支持导出 PDF、结构化数据和正式计分表。 P0 可复现当前确认的正式计分表模板。
FR-11 语音输入 首版通过火山引擎语音识别 API 将语音转成文本，再进入同一 AI 解析链路。 P0 常见比赛口述可被转写，并进入可确认的标准事件流程。
FR-12 房间加入 观众通过四位房间码进入比赛页面。 P0 输入有效四位房间码后进入对应比赛；无效房间码有明确提示。
FR-13 房间生命周期 比赛结束后自动销毁房间。 P0 比赛结束后，原四位房间码不可再进入比赛房间。

8. AI 交互要求

8.1 输入方式

  • 文本输入：直接键入自然语言。
  • 语音输入：通过火山引擎语音识别 API 转写为文本，再进入相同 AI 解析与确认流程。

8.2 支持的首版语句

  • 白队 7 号两分命中
  • 蓝队 12 号犯规一次
  • 白队暂停
  • 开始计时
  • 暂停计时
  • 继续计时
  • 把时间改成 03:12
  • 结束本节比赛
  • 上一条改成白队 11 号得分

8.3 AI 输出字段

字段 说明
队伍 事件归属球队。
球员 若事件与球员有关，则必须明确到人。
事件类型 得分、犯规、暂停、计时控制、修正等。
数据影响 说明本次确认后会如何影响比分和正式统计。
歧义提示 如号码冲突、表述不完整、历史事件引用不清。

8.4 关键约束

  • AI 不直接写入正式数据。
  • 存在歧义时必须先提示，不得自动猜测。
  • 修正历史事件时，要先展示将被修改的原事件。
  • 语音识别失败时，必须允许技术员查看转写文本并手动修正。

9. 数据与输出

9.1 首版正式记分数据

数据类别 字段
比赛信息 赛事名称、日期、时间、地点、裁判、比赛编号。
球队信息 球队名称、球衣颜色、球员姓名、号码、教练信息。
比赛过程 逐分记录、每节得分、暂停、球队犯规、个人犯规、计时控制记录、规则提醒触发记录。
比赛结果 最终比分、胜方、双方队长签字、归档版本。

已定 首版最终输出模板采用当前正式计分表图片对应的格式，产品导出能力必须覆盖其全部关键字段。

  • 可打印 PDF 报告。
  • 可编辑 HTML 文档。
  • 结构化原始数据文件。
  • 正式计分表与签名留底文件。

角色 我想要 从而

12. 异常场景

  • 同队球员号码重复或录入错误。
  • 技术员误记后需要改判上一条事件。
  • 比赛中途修改单节时间。
  • 技术员需要手动更正比赛剩余时间或手动结束本节。
  • 球员第 5 次犯规时需要被及时提醒。
  • 球队单节第 5 次及之后犯规时，需要提醒进入罚球。
  • 比赛剩余 1 分钟时需要倒计时提醒。
  • 网络短时中断后恢复同步。
  • 赛后签字后发现数据需要更正，应生成新版本而不是覆盖旧版。

13. 验收标准

验收项 通过标准
单人可操作 一名技术员可独立完成建赛、记录、纠错、导出。
正式记分完整 导出的正式计分结果可覆盖当前模板中的关键字段。
计时可控 开节前可启动计时；比赛中首屏可直接暂停、继续、更正时间并手动结束本节比赛。
AI 可用 常见文本和语音输入都可稳定转成可确认的标准事件。
暂停规则正确 上半场仅 2 次暂停、下半场仅 3 次暂停、每个加时仅 1 次暂停。
犯规提醒正确 个人第 5 次犯规、球队单节第 5 次及之后犯规、比赛剩余 1 分钟时均能触发提醒。
房间码可用 观众输入有效四位房间码后能进入对应比赛页面。
房间自动销毁 比赛结束后原房间码失效，无法再次进入已结束房间。
签字范围正确 赛后仅要求双方队长签字，并进入最终留底文件。
数据一致 技术员页、观众页、赛后导出结果一致。
可追溯 撤销、补录、修正均有事件记录。
